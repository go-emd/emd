Go-EMD

- Go Language: EMbarrassingly Distributed

Methodologies

Go-EMD contains three major components that are implemented in order to run the distribution successfully. These components are the Worker, Leader, and Connector. With all of these components implemented correctly the distribution will be optimal and DRY.

Leader

There is one and only one leader on each machine in the distribution. This is because one leader handles all REST endpoints and manages each worker on its machine. Upon starting a leader all of its workers will be automatically started. Then the leader will monitor the workers and keep a running cache of each worker' state, health, metrics, and last time updated. Worker

Implements the worker interface and are the routines that do the actual computing within the distribution. Generally speaking a single distribution will consist of a Source (inputs data into the distribution), a Worker of some sort (thought of as a bolt in Apache's Storm), and a Sink (a way of saving data created or minimized).

Above is a diagram showing visually how a worker will work in most cases (dependant on distribution specific implementations). Workers are the go routines that do that actually work/processing on the problem set. Generally speaking each worker will have an input an output and will always have another connection to the leader known as the management connection (MGMT). More on connectors are available below.

Connector

There are a couple types of connectors, and a few more that could be implemented in the future. The purpose of the connector is to simply connect workers to each other. They each implement the connector interface and require a channel for ingress types and a channel for egress types. Below are a list of the current connector implementations and how they work.

  • Local: Translates to a simple go channel of type interface{}. It is meant to allow duplex communication between workers and can transfer any datatype.
  • ExternalIngress: This is more complicated. The ExternalIngress connector is made up of a channel allowing only reading operations. Upon opening the connector it will open a socket for listening on from any host/device. The information sent to it must be in the gob format and the connector will need to be given a type to decode the data received.
  • ExternalEgress: This connector is similar to the ExternalIngress except for the fact that it will only write to the channel and encode the data instead of decoding it. The socket opened in this case will be UDP and only send as well.

NOTE: It is important to note that the gob format is similar to the google buffers encoding. More information can be found here.
